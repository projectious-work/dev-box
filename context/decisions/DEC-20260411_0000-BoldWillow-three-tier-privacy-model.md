spec:
  title: Three-tier privacy model — context/private/ gitignored by aibox
  state: accepted
  decision: v0.16.3 adds `context/private/` and `context/**/private/` to the gitignore
    generated by `update_gitignore` + `ensure_aibox_entries`. Content placed in any
    `private/` subdirectory under `context/` is automatically gitignored (user-private
    tier). The project-private tier is tracked normally. processkit-side questions
    (per-file frontmatter field, docs-site filtering, per-user directory layout for
    multi-user projects) are tracked in `projectious-work/processkit#1` and remain
    open.
  context: Projects need to store personal/sensitive context (API keys in notes, personal
    reflections, team gossip) alongside project-tracked artifacts without risking
    accidental commit. A directory-convention approach is simpler than per-file metadata
    and is already understood by gitignore.
  rationale: The directory convention is observable from the path alone — every tool
    that walks the filesystem already understands it. No schema changes needed in
    aibox. The processkit-side questions (frontmatter field, docs filtering) are deferred
    to processkit, where they belong. aibox's gitignore rule is the minimal necessary
    change.
  alternatives:
  - option: 'Per-file metadata field (e.g. privacy: user-private in YAML frontmatter)'
    rejected_because: Requires schema changes; aibox's stated position is no unless
      a real use case forces it; directory convention is simpler
  consequences: 'Any content in `context/private/` or `context/**/private/` is gitignored.
    Projects that previously tracked private content in non-private paths must migrate.
    processkit issue #1 tracks the remaining open questions.'
