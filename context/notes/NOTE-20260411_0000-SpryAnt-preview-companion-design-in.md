# Preview Companion Container — Design Document

**Date:** 2026-03-26
**Project:** PROJ-004
**Backlog:** BACK-063

---

## Problem Statement

aibox containers use Zellij as the terminal multiplexer. Zellij cannot reliably render images or PDFs — it lacks Kitty image protocol support entirely (Zellij issues #4336/#3981) and its Sixel passthrough is buggy, CPU-intensive, and prone to artifacts. Terminal-based image protocols (Sixel, Kitty, iTerm2 Inline Images) are fundamentally unreliable when running through a multiplexer layer.

Users working with LaTeX documents, technical diagrams (Excalidraw), images, and SVGs need visual preview capabilities that the terminal cannot provide. Current workarounds — chafa/timg character art, entr+mutool watch loops — produce low-fidelity output and add significant image size overhead (ffmpeg alone adds ~450 MB for codec libraries).

## Proposed Solution

A lightweight **companion container** running a web server that serves workspace files with live-reload. The user's browser on the host machine provides pixel-perfect rendering of all visual formats. This sidesteps the terminal image protocol problem entirely — the browser is the display layer, the terminal remains for code and commands.

The companion container runs alongside the main development container, shares the same workspace mount, and is accessed through SSH port forwarding (the same SSH connection that provides terminal access to Zellij).

## Architecture

### Single Companion Container

One container serves all preview needs — not one container per format. This keeps the docker-compose configuration simple and avoids port sprawl.

### Server Component

A small HTTP server (candidates: nginx with lua, a Go binary, a Rust binary, or a Node/Express server) serving:

- **Static files** from the workspace mount at their natural paths
- **WebSocket endpoint** for live-reload notifications
- **Excalidraw editor** at a `/edit/` route prefix
- **pdf.js viewer** for enhanced PDF viewing (zoom, search, page navigation)

A file watcher (inotify-based) monitors the workspace for changes and pushes notifications over WebSocket to connected browsers, triggering automatic page reload.

### Excalidraw Integration

The official `excalidraw/excalidraw` Docker image (~50 MB, nginx:alpine) is a pure client-side SPA with no filesystem integration — files are loaded only via drag-drop or Ctrl+O. To bridge workspace `.excalidraw` files, the companion needs a thin wrapper:

- Embed the `@excalidraw/excalidraw` React component
- A file-bridge endpoint that reads `.excalidraw` files from the workspace and serves them as `initialData` to the component
- A save endpoint that writes modified diagram data back to the workspace file
- The workspace is mounted read-write, so saves persist immediately

### PDF Viewing

Two options, both served by the companion:

1. **Native browser rendering** — navigate directly to the PDF URL; modern browsers render PDFs natively
2. **Embedded pdf.js viewer** — provides enhanced controls (zoom, search, page navigation, presentation mode)

For LaTeX workflows, live-reload is the key feature: `latexmk -pvc` compiles on change, the file watcher detects the updated PDF, and the browser auto-reloads.

### Workspace Mount

The companion container mounts the same workspace volume as the main container (read-write). This means:

- Files edited in the terminal (vim, etc.) are immediately visible to the companion
- Excalidraw saves from the browser are immediately visible in the terminal
- No file synchronization, copying, or API bridging needed

## Workflow Patterns

### LaTeX

1. User runs `latexmk -pvc main.tex` in Zellij terminal
2. User opens `http://localhost:PORT/output/main.pdf` in browser
3. On each save, latexmk recompiles, file watcher detects change, browser auto-reloads
4. Pixel-perfect PDF rendering with zoom, search, and page navigation

### Images and SVG

1. User navigates to `http://localhost:PORT/path/to/file.png` (or `.svg`, `.jpg`, `.webp`, etc.)
2. Browser renders natively at full resolution
3. If the file changes (e.g., regenerated by a build script), live-reload updates the view

### Excalidraw Diagrams

1. User navigates to `http://localhost:PORT/edit/path/to/diagram.excalidraw`
2. Companion serves the Excalidraw SPA with `initialData` loaded from the file
3. User edits in the browser with full Excalidraw functionality (shapes, arrows, text, collaboration-ready format)
4. On save (Ctrl+S or auto-save), the companion writes the updated JSON back to the workspace file
5. The `.excalidraw` file is plain JSON, version-controllable with git

### Yazi Integration

1. User highlights a file in yazi (the terminal file manager)
2. A keybinding sends the file path to the companion server (e.g., `curl http://companion:PORT/open?path=...`)
3. The companion triggers a navigation event over WebSocket
4. The browser navigates to the appropriate viewer for that file type

## SSH Integration

aibox environments are accessed via SSH. The companion container's HTTP port must be forwarded to the user's localhost.

### Port Forwarding

`aibox start` generates or prints an SSH command/config snippet with `-L` port forwards:

```
ssh -L 2222:localhost:22 -L 3030:companion:80 user@host
```

This single SSH connection provides:
- Terminal access (Zellij) on the standard SSH channel
- Web preview access on `http://localhost:3030`

### SSH Config Generation

`aibox start` could also generate an `~/.ssh/config` snippet:

```
Host myproject
  HostName host
  User aibox
  LocalForward 3030 companion:80
```

## Implementation Phases

### Phase 1: Static file server with live-reload

- Companion container with a lightweight HTTP server
- Serves all workspace files at their natural paths
- inotify-based file watcher + WebSocket live-reload
- Covers: images (PNG, JPG, WebP, AVIF), SVG, PDF (via browser native rendering)
- Docker-compose integration (companion service alongside main container)
- Estimated image size: < 20 MB (alpine + static binary or nginx)

### Phase 2: Excalidraw file-bridge

- Embed `@excalidraw/excalidraw` React component (pre-built static assets)
- `/edit/` route prefix serves the Excalidraw SPA
- File-bridge API: GET loads `.excalidraw` JSON as `initialData`, POST saves back
- Auto-save on change with debounce

### Phase 3: Yazi keybinding integration

- Keybinding in yazi that sends the current file path to the companion
- Companion broadcasts navigation event over WebSocket
- Browser navigates to the appropriate viewer
- Configuration in yazi.toml shipped with aibox

### Phase 4: `aibox start` auto-configures SSH port forwarding

- CLI detects companion container port from docker-compose config
- Prints SSH command with `-L` flag for companion port
- Optionally generates SSH config snippet
- `aibox doctor` validates port forwarding is working

## Alternatives Considered

### tmux instead of Zellij

tmux has better terminal image protocol support (Kitty passthrough works in recent versions). However, switching from Zellij would lose:

- Zellij's declarative KDL layout system (core to aibox's layout feature)
- Floating panes and plugin ecosystem
- The user experience investment already made

The companion container approach preserves Zellij while providing superior visual preview quality (browser rendering > terminal image protocols).

### Character-art rendering in terminal

Tools like chafa and timg render images as Unicode block characters or Sixel in the terminal. This works but:

- Low fidelity — fine details are lost
- No interactivity (can't edit Excalidraw diagrams)
- Adds significant image size (ffmpeg ~450 MB, imagemagick ~100 MB)
- Sixel inside Zellij is CPU-intensive and artifact-prone

The companion approach provides pixel-perfect rendering with zero base image bloat.

### X11 forwarding

Running GUI applications (zathura, inkscape) inside the container and forwarding via X11. Not practical because:

- Requires X11 server on the host (not available on macOS without XQuartz, Windows without WSLg/VcXsrv)
- High latency over SSH
- Adds significant container dependencies (X11 libraries, GUI toolkits)
- Does not work in headless/CI environments

## Open Questions

1. **Image size budget:** What is the acceptable size for the companion container image? Target < 50 MB. The Excalidraw SPA assets (~50 MB pre-built) may dominate — should they be fetched on first use instead of baked in?

2. **Server technology:** Options for the HTTP server:
   - **nginx + lua** — proven, small footprint, but lua scripting for WebSocket/file-bridge adds complexity
   - **Go binary** — single static binary, good WebSocket support, fast inotify integration
   - **Rust binary** — aligns with aibox's language, smallest binary size, but build complexity
   - **Node/Express** — natural fit for Excalidraw (same ecosystem), but heavier runtime

3. **Excalidraw assets:** Pre-build the React app into static assets at image build time (deterministic, offline-capable, larger image) vs. serve from CDN (smaller image, requires internet, version drift risk)?

4. **Multiple simultaneous previews:** Should the companion support multiple browser tabs viewing different files? The WebSocket live-reload should be path-scoped (only reload tabs showing changed files), not global.

5. **Security:** The companion serves workspace files over HTTP. In the SSH-forwarded model this is only accessible on localhost — but should there be any authentication layer for environments exposed on a network?

6. **pdf.js vs native:** Is embedded pdf.js worth the complexity, or is native browser PDF rendering sufficient for most users? pdf.js adds search, better zoom control, and presentation mode.
